AV Coders C# Device Library — Code Review

2026-06-10  ·  Full repository: Core, CommunicationClients, 13 driver domains, 12 test projects, packaging & CI  ·  996 tests · 995 passed · 0 failed (2026-06-12)
Five parallel specialist reviews; high-severity findings spot-verified against source before inclusion.

This is a well-architected, professionally packaged library with an unusually good observability story (MEL logging abstraction, diagnostics ring buffers, ActivitySource tracing) and strong test density across driver domains. The risk is concentrated in two places: CommunicationClients has zero tests despite being the most concurrency-heavy code in the repo (and contains several confirmed bugs today), and driver response parsing assumes well-formed, complete messages — malformed or partial input will crash or silently desynchronize drivers.

What's genuinely good

  • Observability built into the foundation — MEL abstraction with no hard logging dependency, per-instance bounded event/error ring buffers with change events, zero-cost ActivitySource tracing with automatic span nesting via PushProperties().
  • Clean driver/transport decoupling — every driver talks through CommunicationClient, so transports are swappable; NoneCommunicationClient is a proper Null Object.
  • TCP reconnection done thoughtfully — exponential backoff with jitter, proactive half-open socket detection via Poll(), careful lock discipline when swapping client/stream pairs.
  • Desired-vs-actual state reconciliation in display drivers — drivers re-send commands when the device reports a state that doesn't match intent.
  • 988 fast tests with excellent parameterization — 478 [Theory]/[InlineData] usages, consistent naming, a shared TestFactory, near-zero flaky patterns.
  • Professional packaging — complete NuGet metadata on all 14 packages (descriptions, MIT license, icon, repo URL, per-package READMEs), nullable enabled everywhere, beta-prerelease CI flow, accurate root README.

Confirmed bugs (re-verified against source during synthesis)

✅ Fixed 2026-06-12Verified AvCodersRestClient.Send(byte[]) sent the literal string "System.Byte[]"
CommunicationClients/AvCodersRestClient.cs:26
bytes.ToString() returned the CLR type name, not the payload — every caller of the byte overload posted garbage to the device.
Resolution: now decodes with Encoding.UTF8.GetString(bytes), matching the UTF-8 string content the Post path sends. Build verified, no new warnings.
✅ Fixed 2026-06-12Verified AvCodersMqttClient ignored its username/password parameters
CommunicationClients/AvCodersMqttClient.cs:16–27
Credentials were accepted but never passed to MqttClientOptionsBuilder — authenticated brokers rejected every connection. The constructor also fire-and-forgot ConnectAsync before attaching the ConnectedAsync handler — a startup race.
Resolution: .WithCredentials(...) is applied when a username is supplied; a null/empty username connects anonymously (the CONNECT packet omits the username/password flags entirely — an empty-string username reads as a failed login to brokers, and MQTT forbids a password without a username). Parameters are now nullable. Handlers are wired before the initial connect, which logs failures and sets Disconnected. The blocking .Wait() in the reconnect loop (H5) remains open.
HighVerified PhilipsSICP.HandleResponse crashes on an empty response
Display/PhilipsSICP.cs:34, 41
response.Length < response[0] dereferences response[0] before any length check — an empty array throws IndexOutOfRangeException. The guard is also off by one, and response[3] is reachable with a 1–3 byte response.
Fix: if (response.Length < 4 || response.Length < response[0] + 1) return; (with the warning log).
Rejected during verification: a reviewer flagged LGCommercial.cs:101 (data[1] after Split("OK")) as a crash risk. It is safe — line 99 returns early unless the response contains "OK", guaranteeing the split yields ≥ 2 parts. Counts on this page reflect verified findings only.

High-severity issues

Transports & threading

High → Partial⚠ Addressed 2026-06-12Zero test coverage for CommunicationClients
Originally: 12 classes implementing reconnection, backoff, send queues and connection-state machines had no test project at all — the highest-risk untested surface in the repo. Now: CommunicationClientsTest exists with 27 tests over loopback sockets/HttpListener covering the TCP client (connect, send/receive, queue-while-disconnected, reconnect-after-drop, refused-connection state), TCP server (accept, receive, probe-byte filtering, multi-client broadcast), UDP client, REST client (incl. the UTF-8 byte-payload regression test) and Wake-on-LAN packet construction.
Remaining: AvCodersSshClient, AvCodersMqttClient, AvCodersSnmpV3Client, AvCodersMulticastClient — these need local server infrastructure or refactoring for dependency injection. Interface and Climate also remain untested.
HighFinalizers call async Stop() — disposal is broken by design
Core/CommunicationClient.cs:205–210 (~IpComms) · Core/ThreadWorker.cs:120–123
A destructor cannot await: the Stop() calls fire-and-forget, finalizer ordering between IpComms and its workers is undefined, and the CancellationTokenSource is never disposed. No concrete client implements IDisposable/IAsyncDisposable.
Fix: implement IAsyncDisposable on IpComms and concrete clients; delete the finalizers.
HighThreadWorker lifecycle races
Core/ThreadWorker.cs:14–26, 125–128
IsRunning reads _cancellationTokenSource/_task without the lock (TOCTOU against StopInternal), and WaitFirst() mutates _waitFirst unsynchronized while the worker loop reads it.
Fix: take _lock in these members, or document them as start-up-only / best-effort.
HighNo TCP framing / partial-read handling
CommunicationClients/AvCodersTcpClient.cs:50–58 · e.g. Matrix/ExtronDtpCpxx.cs:48–65
Each ReadAsync result is treated as one complete message and decoded as ASCII; TCP guarantees neither. Drivers that split on \r lose sync when a frame straddles two reads. CommandStringFormat exists but is never consulted on receive.
Fix: per-connection reassembly buffer with delimiter- or length-based framing; honor CommandStringFormat when decoding.
HighBlocking waits on async paths (3 instances)
AvCodersSshClient.cs:265 · AvCodersMqttClient.cs:62 · Display/SamsungMDC.cs:73–78
SSH reconnect calls Stop().Wait() with no timeout; the MQTT disconnect handler blocks MQTTnet's event pipeline with Task.Delay(3s).Wait() inside a retry loop; SamsungMDC's poll loop uses .Wait(token) instead of await.
Fix: make these paths async end-to-end; if a sync bridge is unavoidable, bound it with a timeout.
HighUDP sends fail silently; unsynchronized client access
AvCodersUdpClient.cs:61, 100
SendAsync isn't wrapped in try/catch — a socket error drops the queued item with no log and no re-queue (TCP re-queues; behavior should match). The _client null-check races with CreateClient() on another thread.
Fix: catch and log/re-queue send failures; guard _client with a lock.

Drivers & core API

HighUnguarded index access into device responses (cross-driver pattern)
NecUhdExternalControl.cs:104–143 · PhilipsSICP.cs:34 · CiscoRoomOs.cs:308–335 · SonySimpleIPControl.cs:120
Malformed or truncated responses throw IndexOutOfRangeException/FormatException and abort the parse. The pattern repeats across roughly 8 drivers: positional access after Split, fixed-offset byte indexing, int.Parse on unvalidated substrings.
Fix: one shared safe-parsing helper (bounds-checked access + TryParse) used by all drivers — a single change that hardens the whole fleet.
HighState-change handler invocations aren't exception-isolated
Core/DeviceBase.cs:26–27 · Core/VolumeControl.cs:31–46 · Conference/CiscoRoomOs.cs:278–284
Legacy delegate fields and modern events are invoked back-to-back with no try/catch — one throwing subscriber prevents the rest from running. CiscoRoomOs rethrows out of its response handler, leaving the driver in an unmarked error state. CommunicationClient.ConnectionState already does this correctly.
Fix: copy the ConnectionState isolation pattern to every handler invocation site.
HighPublic mutable delegate fields instead of events
Core/LogBase.cs · DeviceBase.cs · VolumeControl.cs · Display/Display.cs
Fields like public StringHandler? NameChangedHandlers can be reassigned by any consumer — device.PowerStateHandlers = null silently wipes other subscribers. The codebase is mid-migration to event properties (OnPowerStateChanged).
Fix: finish the migration to event declarations; remove the duplicate legacy fields.
HighLogBase.LoggerFactory swap after construction is a silent no-op
Core/LogBase.cs:21
Each instance captures its ILogger at construction; setting the factory after devices exist changes nothing, and the static setter is unsynchronized.
Fix: resolve the logger lazily through the factory (or document "set once, before constructing any device" and guard with a lock).

Medium-severity issues

13 findings — concurrency, security posture, CI consistency
MediumBiampTtp _currentQuery race
Dsp/BiampTtp.cs:135–149
Checked and cleared from poll and response threads without synchronization — queries can be lost or mismatched. Similar unsynchronized shared state in QsysEcp change-group tracking. Use a lock or Interlocked.
MediumQsysEcp swallows parse exceptions without context
Dsp/QsysEcp.cs:142–145
The exception is logged but not the offending response — include the payload in the log message.
MediumTLS certificate validation disabled unconditionally
AvCodersRestClient.cs:144
ValidateCertificate always returns true. Acceptable on closed AV networks, but it should be opt-in (constructor flag, strict by default) and documented.
MediumAvCodersTcpServer accepts unbounded connections
No client cap or accept throttle — a misbehaving network device can exhaust memory.
MediumSSH: fixed 60 s retry and racy stream creation
AvCodersSshClient.cs:121, 177–194
Reconnect waits a fixed 60 s (TCP starts at 1 s with backoff), and CreateStream is callable concurrently from the receive and connection-check paths with no lock.
MediumNetworkStream I/O not synchronized across threads
AvCodersTcpClient.cs
Assignment of _client/_stream is locked, but concurrent read/write operations from the receive and send-queue workers are not — NetworkStream is not thread-safe for that.
MediumTCP Send uses CancellationToken.None
AvCodersTcpClient.cs:241–280
A hung network blocks the caller indefinitely; bound the write with a timeout token.
MediumNull-forgiving cast in Eaton PDU poll
Power/Eaton.cs:73
x as EatonOutlet followed by outlet!.PollPowerState() — the ! doesn't prevent the NRE. Use if (x is EatonOutlet outlet).
MediumLGCommercial.DoPoll infers power from connection state
Display/LGCommercial.cs:145–171
Connection up ≠ display on — poll the device for power state like SamsungMDC does.
MediumDisposableItems.Dispose stops at the first throwing item
Core/DisposableItems.cs:5–11
Dispose every item and aggregate exceptions, otherwise one bad item leaks the rest.
MediumVolumeControl constructor self-subscribes state-writing lambdas
Core/VolumeControl.cs:54–55
The handler chain re-assigns the backing field the property setter already set — redundant, and confusing about who owns state.
MediumCI version computation duplicated and divergent
.github/workflows/dotnet.yml
The publish-nuget job computes date + run_number while build uses the monthly-counter scheme — versions can drift. Factor into one shared compute-version job. (The workflow edit and both plan files are currently uncommitted.)
MediumParsing edge cases untested everywhere
Across 988 tests there is essentially no malformed/empty/truncated-response coverage — exactly the input class behind the high-severity parsing crashes. Parameterized "garbage in" tests per driver are cheap and high-value.
